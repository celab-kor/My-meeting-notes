01
서류 제출 업무 논의

논의 배경

박규호 부장이 전달받은 경적서 및 관련 서류의 제출 방식에 대해 확인이 필요했습니다. 이미 심의를 통과한 내부 결제용 문서로, 정확한 양식과 내용 확인이 중요한 상황이었습니다.

주요 논의 내용

경적서 제출 확정
• 금액: 480만원으로 확정
• 이전에 전달받은 경적서를 그대로 사용
• 심의 통과 완료된 문서로 부담 없이 진행 가능
산출의용서 작성 방식
• 자유형식으로 지정된 서류
• 경적서의 테이블 형식을 그대로 활용
• 제목만 "산출의용서"로 변경하여 제출
기타 제출 서류
• 참여이약 현황표: 박규호 부장만 기재
• 참여이약 재직증명서: 서식 청구 가능
• 보안서약서: 표준 서식 활용

Insight

  • 절차적 효율성: 이미 심의를 통과한 문서를 활용함으로써 재검토 리스크를 제거하고 행정 절차를 간소화했습니다.
  • 양식 유연성 활용: 자유형식이라는 지침을 활용하여 기존 검증된 포맷을 재사용하는 실용적 접근을 취했습니다.
  • 최소 참여 인력: 현 단계에서는 부장 1명만 참여이약에 포함하여 프로젝트 규모를 적정하게 유지했습니다.
02
홈페이지 제작 핵심 목적

제1 목표: 제품 소개 및 데모

핵심 가치 제안:

화공 플랜트 설계를 도와주는 툴이라는 제품의 특성상, 엔지니어와 의사결정자들이 웹사이트를 방문했을 때 "매력적이고 신뢰할 수 있는 솔루션"이라는 인상을 즉시 받아야 합니다.

이를 위해 반응형 인터랙티브 데모가 필수적이며, 실제로 툴이 작동하는 것처럼 보이는 애니메이션이 핵심 요구사항입니다.

레퍼런스 사이트: 알칼리 엔지니어링
• 반응형 에셋이 구현된 모범 사례
• 채팅 버튼 클릭 시 애니메이션으로 결과 표시
• 실제 소프트웨어처럼 보이는 UX 제공
• 영상이 아닌 JavaScript 애니메이션 활용으로 빠른 로딩과 높은 전환율 확보

구현 방식: 영상 삽입 대신 JSON/JavaScript 기반 애니메이션을 사용합니다. 이는 사용자 이탈율을 낮추고, 즉각적인 피드백을 제공하며, 페이지 로딩 속도를 최적화하는 전략입니다.

제2 목표: 회사 소개 및 신뢰도 구축

발표회나 미팅에서 회사 웹사이트 존재 여부를 자주 확인받는 상황을 고려하여, 회사의 정체성과 신뢰도를 보여줄 수 있는 페이지가 필요합니다.

회사 소개 페이지 구성 요소
• 협력 업체 및 파트너사 정보
• 협업 엔지니어 프로필
• 수상 경력 및 인증
• 컨택트 정보 (이메일 수집 폼)
• 회사 비전 및 미션

Insight

  • 이중 목표 구조: B2B 시장에서 제품력과 회사 신뢰도는 동등하게 중요합니다. 두 요소를 명확히 분리하여 각각 최적화하는 전략이 효과적입니다.
  • First Impression 설계: 엔지니어링 업계 특성상 기술적 완성도와 전문성을 시각적으로 즉시 전달해야 합니다.
  • 리드 생성 메커니즘: 제품에 관심 있는 방문자를 잠재 고객으로 전환하는 명확한 경로 제공이 핵심입니다.

기술 선택 근거
React vs Next.js:
• React: 기본적인 반응형 UI 구현 가능
• Next.js 선택: SEO 최적화가 비즈니스 성장에 필수적
• 검색 엔진에서 솔루션 페이지가 상위 노출되어야 함

영상 vs JavaScript 애니메이션:
• 영상 삽입: 클릭 전환율 감소, 로딩 시간 증가
• JS 애니메이션 선택: 즉각적 반응, 가벼운 용량, 높은 참여율
• 사용자가 직접 조작하는 듯한 경험 제공

핵심 인사이트 및 전략적 시사점

🎯 전략적 인사이트

1. UX 중심의 전략적 설계

회사 소개를 네비게이션 바에서 제외하고 하단 로고 클릭으로만 접근 가능하게 설계한 결정은 의사결정 장벽을 최소화하는 성장 해킹 전략입니다.

대부분의 B2B 웹사이트가 "회사소개", "제품", "고객사례", "문의하기" 등 모든 정보를 네비게이션에 나열하지만, 스냅스케일은 "제품에만 집중하게 하는" 파격적 선택을 했습니다.

이는 Hick's Law(선택지가 많을수록 결정 시간이 증가한다)를 적용한 것으로, 사용자가 제품 가치를 먼저 경험한 후 회사 정보가 궁금해질 때 찾아볼 수 있도록 설계한 전략입니다.

비즈니스 임팩트: 잠재 고객이 제품 데모를 보는 비율(전환율)을 극대화하고, "이 회사가 믿을만한가?"라는 질문에 대한 답은 이미 제품에 관심을 가진 사람에게만 제공합니다.

2. 기술 부채 관리의 현실적 접근

"깔끔한 코드보다 빠른 결과물 우선"이라는 명시적 합의는 스타트업의 시간 압박과 자원 제약을 정직하게 반영한 결정입니다.

많은 프로젝트가 "완벽한 코드"를 추구하다가 출시 시기를 놓치는 반면, 스냅스케일은 "작동하는 MVP를 먼저 확보"한 후 점진적 개선을 추구하는 린(Lean) 접근법을 채택했습니다.

특히 대표가 게임 개발 경험이 있어 코드 이해가 가능하다는 점을 고려하여, 극도로 추상화된 코드보다는 명확하고 직관적인 코드를 선호했습니다. 이는 장기적으로 외주 의존도를 낮추고 자립 가능성을 높이는 전략입니다.

리스크 관리: 나중에 리팩토링 가능성을 열어두되, 12월 7일 심의 마감을 준수하는 것이 최우선 목표입니다. "완벽한 코드로 늦게 출시"보다 "작동하는 코드로 제때 출시"가 스타트업에게 더 중요합니다.

3. 인터랙티브 데모의 심리적 효과

실제 소프트웨어가 작동하는 것처럼 보이는 JavaScript 애니메이션 구현은 복잡한 백엔드 없이도 제품 가치를 효과적으로 전달할 수 있는 방법입니다.

"런(Run)" 버튼을 누르면 단계별로 이미지가 전환되며 엑셀 파일이 완성되는 과정을 보여주는 것은 실제 소프트웨어 작동이 아니지만, 사용자는 "이 툴이 실제로 작동한다"는 인식을 갖게 됩니다.

특히 한 번만 입력 가능하게 제한한 설계는 "실제 제품 체험"이라는 진정성을 강화하고, 사용자가 신중하게 입력하도록 유도합니다.

영상 vs 애니메이션 선택의 근거:
• 영상: 클릭해야 재생, 로딩 시간, 낮은 참여율
• 애니메이션: 즉각 반응, 가벼운 용량, 높은 몰입감

이는 사용자 이탈율을 낮추고 "이 툴을 써보고 싶다"는 욕구를 극대화하는 전략입니다.

4. 교육과 협업의 이중 구조

GitHub Pages 실습을 통해 비개발자(부장)에게 기술적 이해를 제공하면서도, 전문 개발자에게 외주를 맡기는 하이브리드 접근법은 장기적 자립 가능성과 단기 효율성의 균형을 추구합니다.

"모든 것을 직접 하기"와 "모든 것을 외주 맡기기" 사이의 최적점을 찾는 것이 스타트업의 핵심 과제입니다. 스냅스케일은 핵심 역량만 내재화하고 나머지는 외부 전문가를 활용하는 전략을 선택했습니다.

특히 "대소문자 구분", "폴더 구조" 등 개발자에게는 당연한 개념이 비개발자에게는 큰 장벽임을 인식하고, Canvas 강의 제작 등 체계적인 교육 시스템 구축을 계획했습니다.

장기적 효과: 간단한 콘텐츠 수정은 내부에서 처리하고, 복잡한 기능 개발은 외주를 활용함으로써 운영 비용을 최적화할 수 있습니다.

5. SEO와 사용자 여정의 전략적 분리

3개의 주요 진입점(SimuLie, AutoFlow, 회사 메인)을 설정하되, 솔루션 페이지에 SEO를 집중하는 전략은 검색 엔진 최적화와 전환율 최적화(CRO)를 동시에 고려한 설계입니다.

대부분의 잠재 고객은 "화공 플랜트 시뮬레이션", "프로세스 자동화 설계" 같은 문제 해결 키워드로 검색할 것입니다. 회사명으로 검색하는 경우는 이미 회사를 알고 있는 경우이므로, SEO 리소스를 솔루션 페이지에 집중하는 것이 효율적입니다.

사용자 여정 설계:
1. 검색 → 솔루션 페이지 유입
2. 제품 데모 체험 → 가치 인식
3. 이메일 입력 (Join List or Request Demo)
4. 궁금한 경우 하단 로고 클릭 → 회사 소개

이는 사용자가 제품 가치를 먼저 경험한 후 회사에 대한 신뢰를 쌓는 순서를 따릅니다.
